# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Current state of this repository

The app lives in **`proyecto-trayectoria/`** (React + Vite), named **"De la Raíz a la Huella"**. It has been scaffolded end-to-end (Fases 1–4 of the roadmap) and smoke-tested with Playwright — see `proyecto-trayectoria/README.md` for setup/run/deploy commands and the current pending-items list (incomplete raw materials for some professionals — the 3 reflection questions and activity name are resolved).

The repo root also holds:
- **`plan_actividad_evaluando_trayectoria.md`** — the full functional spec (source of truth for behavior/data model/API contract). Re-read it before making structural changes to the app.
- **`Public/Profesionales/<Nombre Completo>/`** — the raw, unprocessed materials (bios in `.docx` + stage photos) that `proyecto-trayectoria/scripts/extraer-textos.js` consumes. Do not edit these in place; the script copies/normalizes into `proyecto-trayectoria/public/profesionales/`.

## Commands (run from `proyecto-trayectoria/`)

- `npm run dev` — dev server (Vite picks a free port starting at 5173 if others are in use — check its stdout for the actual port).
- `npm run preparar-datos` — regenerate `public/profesionales/profesionales.json` and the per-person image folders from `../Public/Profesionales`. Re-run whenever raw materials change.
- `npm run build` — production build to `dist/`.
- `npm run deploy` — build + publish `dist/` to the `gh-pages` branch.

There is no test suite (no automated tests were requested); verification so far has been a manual/Playwright smoke pass of both routes, including keyboard-driven navigation through the full slide sequence.

## Architecture

- **Frontend**: React + Vite, `HashRouter` (needed for GitHub Pages static hosting), `base: './'` in `vite.config.js` so it doesn't depend on the repo name.
  - `/#/` → `Formulario.jsx` — participant form (name selector + 3 questions from `src/data/preguntas.js`, sourced from `src/data/momentos.js`), POSTs via `src/services/sheetsApi.js`. Restyled to match the presentation's dark/animated look (shares `FondoAnimado.jsx`) since participants see it as part of the same show, not a disconnected utility page.
  - `/#/presentacion` → `Presentacion.jsx` — the entire show, single window, no facilitator panel. Deliberately built as a highly animated "show" (framer-motion + canvas-confetti), not a plain content page — the user explicitly asked for something dynamic/eye-catching/interactive to present live in front of the team, not just a functional viewer.
- **There is no control panel and no `BroadcastChannel`/two-tab sync anymore** (that was the first iteration; the user explicitly rejected it: "no quiero que haya panel de control"). `Presentacion.jsx` is fully self-contained: it holds `indiceSlide` state and a linear `secuencia` array (built by `construirSecuencia`), advanced by keyboard (→/space/click = next, ← = previous) via a `window` `keydown` listener. If you're tempted to reintroduce a second window/tab for facilitator controls, don't — that was a deliberate reversal, not an oversight.
- **Slide sequence** (`construirSecuencia` in `Presentacion.jsx`): `[bienvenida, agenda, evidencias, oracion, reflexion, qr, ...for each person: Inicio, Medio, Actual, Final slides..., evaluacion]`. **People are ordered by their response `Timestamp`, ascending — not chosen by a facilitator.** Anyone without a submitted response yet is appended at the end in `profesionales.json` order, so nobody is dropped from the show even if they haven't answered when the presentation is opened. This requires `apps-script/Code.gs`'s `doGet` to expose `timestamp` per response (it does) and `doPost` to preserve the *original* timestamp on resubmission rather than resetting it (otherwise correcting a typo would reshuffle someone's place in the show).
- **QR slide** (`SlideQR` in `Presentacion.jsx`): generated client-side with the `qrcode` package (no external API call — matters for a live event on unreliable venue wifi). The encoded URL is computed from `window.location.origin` + `pathname` + `'#/'` at render time, never hardcoded, so it's correct whether running on `localhost` or the eventual GitHub Pages domain.
- **The 4 "momentos"** (`src/data/momentos.js`, no emojis — removed per explicit request): Mis raíces (Inicio — answered by `textoInicio` from the `.docx`), Mis decisiones (Medio), Mi propósito (Actual), Mi legado (Final). Each stage shows just the momento's own title (no "Momento N ·" prefix — removed per explicit request), rendered uppercase via `.presentacion-momento-titulo--mayuscula`, plus **both the question and the answer** (not just the answer, and the question is styled larger/bolder than the answer) — `preguntas.js` derives the 3 form questions from this same file, so it's the single source of truth used by `Formulario.jsx` and `Presentacion.jsx`.
- **Surrounding agenda content** (`src/data/agenda.js`): `BIENVENIDA` (opening welcome message), `AGENDA` (the 4-point agenda list — point 3 uses `NOMBRE_ACTIVIDAD`, not a generic label), `EVIDENCIAS_APRENDIZAJE` (bullets Claude generated from the activity's overall design — flagged in the README as something the user should review/edit, since the user didn't dictate this text directly), `ORACION.texto` (extracted once from `../Oración.docx` via mammoth and hardcoded — it's static, no need to parse it at runtime), and `REFLEXION.video` (a filename served straight from `public/Reflexión.mp4` via a native `<video controls>` — not an external YouTube/Drive embed; `SlideReflexion` in `Presentacion.jsx` `encodeURIComponent`s the filename since it has an accented character. `null` shows "Video pendiente" as a fallback). `TOTAL_EQUIPO` (18, from the plan) and `UMBRAL_REFLEXION_PROFUNDA` (a length-based heuristic, tune if needed) drive the closing `Evaluación` slide's two stats (participation %, "reflexiones profundas" %) — see plan section 11. Punto 4's actual dynamic is still undefined in the plan (section 14); these computed indicators are the current stand-in.
- **Backend**: Google Apps Script Web App (`apps-script/Code.gs`) bound to a Google Sheet, `doPost`/`doGet` on sheet `Respuestas`. POST requests are sent without a `Content-Type: application/json` header on purpose (defaults to `text/plain`) to avoid a CORS preflight that Apps Script doesn't handle — `Code.gs` still parses the body as JSON regardless.
- **Duplicate submissions**: if someone resubmits the form, `doPost` overwrites their existing row (matching `Nombre`) but **keeps the original Timestamp** — this was an open decision in the plan (section 6.1), resolved this way because the timestamp now also drives presentation order, not just bookkeeping.
- **Data model**: `public/profesionales/profesionales.json` (`{ nombre, textoInicio, carpeta, imagenes: { Inicio, Medio, Actual, Final } }`, only the stages that actually have an image are present) + Google Sheets `Respuestas` (now including `timestamp`) for the 3 live answers.
- **Non-negotiable business rule**: the person's name is only rendered in the DOM from the "Actual" stage onward (`Presentacion.jsx` conditionally renders the `<h2>` at all, not just CSS-hidden) — verified via Playwright by inspecting DOM HTML at each stage, not just visually.
- **Missing data handling**: missing photo → "Foto pendiente" placeholder; missing form response → "Respuesta pendiente" placeholder. The UI must not break when either is absent (several professionals currently have incomplete raw materials — see README's pending-items list).

## Raw materials: known irregularities

`Public/Profesionales/` (17 folders, not the 18 the plan assumes) is **not uniform** — `extraer-textos.js` already handles this, but if you touch that script, know the shape of the input:

- Raw materials live at `Public/Profesionales/` (capital letters) at the repo root, outside the Vite project — not to be confused with `proyecto-trayectoria/public/` (lowercase, Vite-served). **This is the single source of truth for the data-prep pipeline** — don't add photos directly into `proyecto-trayectoria/public/profesionales/` (the generated folder); they won't be registered in `profesionales.json` and will just sit there orphaned until reconciled by hand (this happened once — see git history / this file's earlier revisions for the cleanup). Always add materials to the raw folder and re-run `npm run preparar-datos`.
- Stage image basenames are inconsistent: `Medio.jpeg` vs `Media.jpeg` for the same stage (the script normalizes both to `Medio`), and extensions vary (`.jpg`/`.jpeg`/`.png`).
- `.docx` filenames don't match folder/person names — match people by folder name. If there's no `.docx` at all, a plain `bio.txt` in the person's folder also works (added to support Alexander de Jesús Ossa Calvo, whose bio arrived as pasted text, not a document) — `extraer-textos.js` tries `.docx` first, then falls back to `.txt`.
- All 17 people now have complete photo sets (Inicio/Medio/Actual/Final) and bio text, **except Yeison Suárez Suárez**: his `.docx` exists but is textually empty — needs a real bio (docx or `bio.txt`) before his "Mis raíces" momento shows anything but "Respuesta pendiente". The plan expects 18 professionals; only 17 have a folder at all — one person is still missing entirely.
- **The bios start by restating the person's own name** (e.g. "Carolina Gallego Desde pequeña soñé...", "Cristian Heli Flórez Era un niño..." — note spelling varies from the folder name). `extraer-textos.js` strips this leading name mention (fuzzy/typo-tolerant match, capped at 4 words) before it becomes `textoInicio` — otherwise the name would visibly leak on screen during the "Inicio"/"Medio" stages even though it's never rendered as a separate DOM element, defeating the hidden-name business rule. If you touch that stripping logic, re-verify by printing every person's `textoInicio` — it's not exact-match (docx and folder spellings differ) and a too-loose or too-strict distance threshold either leaks the name or eats real bio content.
- `Oración.docx` at the repo root **is used** — its text was extracted once (via mammoth) into `ORACION.texto` in `src/data/agenda.js`. It's not re-parsed at runtime, so if the prayer text changes, re-extract and update that constant by hand.
